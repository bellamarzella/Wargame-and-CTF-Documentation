# Natas Levels 5 → 11
| Level | Vulnerability | Cause | Action | Remediation |
| :--- | :--- | :--- | :--- | :--- |
| **5 → 6** | Broken Authentication (Cookie Manipulation) | Relying on a client-side cookie value (`loggedin=0`) to determine authorisation status without server-side validation. | Intercepted the HTTP request in **Burp Suite**, modified the `loggedin` cookie value from `0` to `1`, and forwarded the request to reveal the password. | Never trust client-side state for authentication or authorisation; track authorisation states securely on the server side (e.g., in a secure session database). |
| **6 → 7** | Sensitive File Exposure (Source Code Disclosure) | Storing sensitive configuration secrets in a web-accessible directory (`includes/secret.inc`) with default read permissions. | Identified the `include` file path in the source, navigated directly to `includes/secret.inc` in the browser, read the `$secret` value, and submitted it to get the password. | Store all backend configuration files, secrets, and environment variables outside of the web server's public document root. |
| **7 → 8** | Local File Inclusion (LFI) | Allowing user input to dictate which local file is included without proper sanitisation or validation. | Exploited the `page` parameter in the URL to include `/etc/natas_webpass/natas8`, revealing the password. | Validate and sanitise all user input, and avoid including files based on user-supplied data. Use whitelisting for allowed files. |
| **8 → 9** | Insufficient Cryptographic Obfuscation | Relying on reversible encoding functions (`bin2hex`, `strrev`, `base64_encode`) instead of robust encryption to protect sensitive data. | Analysed the exposed PHP source code, identified the exact sequence of encoding functions used on the secret, and reversed the process (Hex to ASCII → Reverse → Base64 Decode) to recover the original cleartext secret. | Never use simple encoding or obfuscation algorithms to secure sensitive data. Use industry-standard, cryptographically secure hashing (e.g., bcrypt) or encryption (e.g., AES). |
| **9 → 10** | OS Command Injection | Passing unsanitised user input directly into a system shell execution function (`passthru()`), allowing shell metacharacters to alter the intended command. | Injected a semicolon `;` to terminate the original server-side `grep` command, followed by a `cat` command to read the target password file (`/etc/natas_webpass/natas10`). | Avoid executing system shell commands from application code. If unavoidable, strictly sanitise inputs, escape arguments using `escapeshellarg()`, or use built-in language APIs instead of shell execution. |
| **10 → 11** | Command Injection Bypass (Newline Injection) | Blacklisting specific shell metacharacters (like `;` and `&`) while failing to sanitise or filter raw control characters like newlines. | Bypassed the character blacklist by injecting URL-encoded newline characters (`%0A`), forcing the underlying shell to treat the input as separate, sequentially executed command lines and read `/etc/natas_webpass/natas11`. | Use robust, whitelist-based input validation instead of blacklisting dangerous characters, or avoid passing raw user parameters to shell functions altogether. |
