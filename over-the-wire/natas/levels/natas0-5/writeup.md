# Natas Levels 0 → 5
| Level | Vulnerability | Cause | Action | Remediation |
| :--- | :--- | :--- | :--- | :--- |
| **0 → 1** | Plaintext Information Disclosure | Storing sensitive credentials in client-side HTML comments. | Opened browser **DevTools (F12)** to inspect the HTML source and read the commented-out password. | Remove all sensitive comments and credentials before deploying to production. |
| **1 → 2** | Client-Side Security Bypass | Relying on client-side JavaScript restrictions to prevent users from viewing source code. | Bypassed the disabled right-click restriction by pressing **F12** directly (or prefixing the URL with `view-source:`) to inspect the source. | Never rely on client-side controls for security; design systems assuming the user has total control of the browser. |
| **2 → 3** | Unprotected Directory Listing | Web server misconfiguration that allows directory indexing when an `index` file is missing. | Spotted an asset loaded from `/files/pixel.png`, navigated directly to `/files/` in the URL bar, and read the exposed `users.txt` file. | Disable directory listings in the web server configuration (e.g., set `Options -Indexes` in Apache). |
| **3 → 4** | Information Disclosure via `robots.txt` | Treating `robots.txt` as a security mechanism to hide folders rather than an advisory file for web crawlers. | Visited `/robots.txt` to check search engine exclusion rules, discovered the "hidden" `/s3cr3t/` directory, and accessed the password file. | Use robust server-side authentication and folder-level access controls to protect sensitive files rather than obscurity. |
| **4 → 5** | Weak Access Control (HTTP Headers) | Trusting and validating client-supplied HTTP headers (like `Referer`) to authenticate the request origin. | Intercepted the HTTP request using **Burp Suite** and spoofed the `Referer` header to match the expected URL. | Implement secure, cryptographically signed server-side session tracking to manage authentication. |
